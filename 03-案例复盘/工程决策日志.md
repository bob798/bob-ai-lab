## ## 0310 · 从一个 Bug 出发，重建需求管理体系

---

**起点：一个功能被遗漏了**

V0.2b 开发中，发现历史会话没有「查看复盘」入口。按正常流程这应该是个 Bug，但深挖根因后发现：代码没有错，是需求从文档到执行指令的转化过程中，场景悄悄丢失了。

---

**第一层思考：为什么场景会丢失**

原来的流程是：Human 描述功能 → Claude 写 SPEC → Claude 写执行指令。

中间那次转化完全靠人脑记忆。SPEC 里写了"从历史记录进入"这个场景，但转化成执行指令时漏掉了，没有任何机制能拦截这个错误。

结论：**不是执行问题，是层级缺失问题。**

---

**第二层思考：怎么从结构上杜绝**

引入 BDD（行为驱动开发）作为中间层：

```
User Story（用户要什么）
    ↓
BDD Scenarios（每个场景显式列出）  ← 新增的防漏层
    ↓
执行指令（从 Scenarios 直接生成）
```

BDD 的核心机制是强制反问：**"用户还有哪种路径能达到同一个目标？"** 这个问题一问，"从历史记录进入"这条路径就不会被漏掉。

---

**第三层思考：文档管理本身也需要管理**

随着项目代码量增长，出现了更大的问题：

- **决策遗忘**：为什么选 SQLite？为什么 Alex 不纠错？三个月后连自己都忘了
- **逻辑碎片化**：每个新的 AI 实例都要花大量 token 重建上下文
- **原型与实现漂移**：没人知道文档和代码哪里不一致了

对应三个解法：ADR 沉淀决策、C4 Model 可视化架构、Living Documentation 让文档跟着代码走。

---

**最终落地的体系**

```
Human 的工作前移：
  以前 → 写详细 PRD，描述怎么实现
  现在 → 用自然语言说想法，确认场景完整性（15分钟/版本）

AI 的职责扩展：
  以前 → 生成代码
  现在 → 需求整理 + 场景推导 + 完整实现 + 文档同步

防漏机制：
  场景确认前置 → BDD Scenarios 写在代码之前
  决策永久沉淀 → ADR 记录每个重要判断
  状态自动同步 → Claude Code 每个 Step 后自动更新文档
```

---

**核心洞见**

> 传统方法论的"需求文档"是给人写的，目的是让团队对齐。 AI 协作场景下，文档的作用变了——它是给下一个 AI 实例的上下文，是防止 AI 遗忘和漂移的锚点。 文档管理方式不变，只是换了主要读者。

---

**关于图中的深度思考点**

"未来非技术人员如何查看系统规则而不依赖开发人员"——这个问题今天也有了答案：

ROADMAP.md 记录做了什么、不做什么；ADR 记录为什么这么决策；HUMAN_PLAYBOOK.md 是操作手册。这三个文档加在一起，任何人打开就能理解系统的现状和背后的逻辑，不需要问工程师。
